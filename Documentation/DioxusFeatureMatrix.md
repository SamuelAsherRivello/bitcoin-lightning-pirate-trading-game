# Dioxus Feature Matrix

Keep this file updated as development continues. When routes, platform support, cache behavior, or Dioxus feature usage changes, update the matching rows in the same change.

| # | Feature | Status | Current template usage | Docs |
| - | ------- | ------ | ---------------------- | ---- |
| 1 | Components | ✅ | Shared UI uses `#[component] fn Name(...) -> Element` in components such as [`Page`](../packages/ui/src/client/components/page.rs), [`PageHeader`](../packages/ui/src/client/components/page_header.rs), [`PageFooter`](../packages/ui/src/client/components/page_footer.rs), setup components such as [`WarningCallout`](../packages/ui/src/client/components/setup/mod.rs), game history components such as [`HistoryItems`](../packages/ui/src/client/components/game/mod.rs), and routed pages such as [`Home`](../packages/ui/src/client/pages/home.rs), [`SetUp`](../packages/ui/src/client/pages/setup.rs), [`PlayGame`](../packages/ui/src/client/pages/play_game.rs), and [`DebugNetwork`](../packages/ui/src/client/pages/debug_network.rs). | [Components](https://dioxuslabs.com/learn/0.7/essentials/ui/components/) |
| 2 | RSX markup | ✅ | Pages and components render HTML-like UI through `rsx!`; the active lab pages use direct RSX loops for Home FAQ rows, the Bitcoin vs Lightning comparison table, setup checklists, route rows, recent-action history detail pills, invoices, and payments. | [RSX](https://dioxuslabs.com/learn/0.7/essentials/ui/rsx/) |
| 3 | Signals | ✅ | [`AppLayout`](../packages/ui/src/client/mod.rs) provides theme, language, setup profile, lab state, toast, and blocking operation-prompt signals. Lab pages use local signals for forms, busy states, prompt/toast sequence ids, and AutoSend toggles. | [Signals](https://dioxuslabs.com/learn/0.7/essentials/basics/signals/) |
| 4 | Context state | ✅ | The app uses `use_context_provider` and `use_context::<Signal<T>>()` for shared UI state, including subtle happy-path toasts and conspicuous broken/reset prompts. | [Global context](https://dioxuslabs.com/learn/0.7/essentials/basics/context/) |
| 5 | Router | ✅ | [`Route`](../packages/ui/src/client/mod.rs) maps `/` to `Home`, `/setup` to `Set Up`, `/play` to `Play Game`, and `/debug` to `Network Dashboard`. | [Routes](https://dioxuslabs.com/learn/0.7/essentials/router/routes/) |
| 6 | Router links | ✅ | [`PageHeader`](../packages/ui/src/client/components/page_header.rs) uses router-aware `Link` components for `Home` and `Set Up`, plus enabled links or disabled button affordances for locked lab routes. | [Navigation](https://dioxuslabs.com/learn/0.7/essentials/router/navigation/) |
| 7 | Asset macro | ✅ | Entry CSS, app-shell CSS, component CSS, and flag images use `asset!("/assets/...")`; the shared [`App`](../packages/ui/src/client/app.rs) links header/footer styles before the routed shell renders to reduce first-paint flicker. | [Assets](https://dioxuslabs.com/learn/0.7/essentials/ui/assets/) |
| 8 | Async actions | ✅ | Lab pages use async event handlers and `spawn` inside `use_effect` to load setup/lab state, verify the Polar bridge before step 1 passes, ensure the named Polar server exists, create/destroy Polar demo nodes, recover stale Polar lab state, open routes, mine blocks, create invoices, and pay invoices without blocking render. Polar setup resets now rewind only local app state, while Polar bridge/server/node work stays on submit actions; happy-path operations report through toasts and broken-lab operations use the blocking prompt surface. | [Async](https://dioxuslabs.com/learn/0.7/essentials/basics/async/) |
| 9 | Effects and timers | ✅ | [`AppLayout`](../packages/ui/src/client/mod.rs) validates and refreshes saved connected Polar setup on every primary route, then polls every second so externally mined Polar blocks update the shared lab state; lab pages use `use_effect` for route-entry lab-state hydration and Polar health validation, and [`toast`](../packages/ui/src/client/components/toast.rs) owns platform-specific toast timeout plus 0.25 second minimum prompt-message timing. | [Effects](https://dioxuslabs.com/learn/0.7/essentials/basics/effects/) |
| 10 | Browser local storage | ✅ | [`storage_service`](../packages/ui/src/client/services/storage_service.rs) stores browser theme, language, setup profile, demo lab state, and template data snapshots in localStorage. | [Web platform](https://dioxuslabs.com/learn/0.7/guides/platforms/web/) |
| 11 | Native SQLite cache | ✅ | [`database_service`](../packages/ui/src/client/services/database_service.rs) reads native SQLite data under `data/` for non-wasm builds. | [Async](https://dioxuslabs.com/learn/0.7/essentials/basics/async/) |
| 12 | First-time DB creation | ✅ | [`create_database_if_missing()`](../packages/ui/src/client/services/database_service.rs) is the boilerplate location for first-time native schema creation and seed-row insertion. Normal reads do not recreate or reseed an existing DB. | [Async](https://dioxuslabs.com/learn/0.7/essentials/basics/async/) |
| 13 | Localization | ✅ | [`localization_service`](../packages/ui/src/client/services/localization_service.rs) loads Fluent bundles for English, Spanish, Portuguese, and French. | [Internationalization crate](https://crates.io/crates/dioxus-i18n) |
| 14 | Theme and setup persistence | ✅ | [`storage_service`](../packages/ui/src/client/services/storage_service.rs) persists theme, language, setup profile values, and demo lab state for web and desktop. | [Signals](https://dioxuslabs.com/learn/0.7/essentials/basics/signals/) |
| 15 | Web and desktop targets | ✅ | [`packages/web`](../packages/web/src/main.rs) and [`packages/desktop`](../packages/desktop/src/main.rs) launch the same shared UI; the root [`index.html`](../index.html) and desktop WebView config use the default dark shell color before the app renders. | [Platforms](https://dioxuslabs.com/learn/0.7/guides/platforms/) |
| 16 | Server contact probe | ✅ | [`DeveloperTools`](../packages/ui/src/client/components/developer_tools.rs) includes a Server button that probes the same-origin web server and reports success or failure through the shared toast. | [Web platform](https://dioxuslabs.com/learn/0.7/guides/platforms/web/) |
| 17 | Forms | ✅ | [`SetUp`](../packages/ui/src/client/pages/setup.rs) uses the `Setup` page heading with `General` and `Connection` second-level sections. `General` owns `Sats per transaction`; `Connection` owns the `Polar Connection (Networked)` / `Mock Connection (Offline)` tabs. The Polar tab is split into `OS` and `Polar` third-level sections for install/run guidance and compact form rows for bridge URL, editable server name, demo-node creation, and final completion; reset buttons rewind the four-step wizard state locally, with step 4 still requiring an "Are you sure?" prompt before locking the app routes. | [Forms](https://dioxuslabs.com/learn/0.7/essentials/fullstack/forms/) |
| 18 | Server-operation boundary | ⚠️ | [`lightning_server_functions`](../packages/ui/src/client/services/lightning_server_functions.rs) mirrors the planned server-function contracts with local async wrappers for the current web/desktop POC state path, including Polar bridge calls for server existence/create/start checks, stopped-network validation before Bitcoin RPC/mining, demo-node lifecycle, and health recovery that locks stale gameplay/debug state. Dioxus `#[post]` macro wiring is deferred because the current `0.7.7` server feature resolves an unavailable `dioxus-liveview = 0.7.7` package in this environment. | [Server functions](https://dioxuslabs.com/learn/0.7/essentials/fullstack/server_functions/) |
| 19 | Lightning service crate | ✅ | [`packages/lightning-service`](../packages/lightning-service/src/lib.rs) exposes client-safe domain DTOs and lab state operations from [`src/client`](../packages/lightning-service/src/client/mod.rs), while local profile loading, the optional `tonic_lnd` adapter boundary, and native/server lab observer event types live under [`src/server`](../packages/lightning-service/src/server/mod.rs). | [Fullstack](https://dioxuslabs.com/learn/0.7/essentials/fullstack/) |
| 20 | Polar health recovery | ✅ | [`polar_bridge_service`](../packages/ui/src/client/services/polar_bridge_service.rs) classifies bridge, network, Bitcoin backend, and demo-node health from Polar's request/response bridge; step 3 setup updates the blocking prompt for each node while it reuses only exact lowercase app-owned nodes with the exact fresh demo wallet balance, destroys/recreates stale demo nodes, starts, funds, mines, and wallet-checks Alice, Bob, and Carol before confirming success; every primary route polls connected Polar state, external block-height increases activate pending trade routes, and broken lab state shows a blocking prompt, locks stale connected state, and routes back to `Home`. | [Async](https://dioxuslabs.com/learn/0.7/essentials/basics/async/) |
| x3 | Dynamic routes | ❌ | The template currently has only static routes. Add dynamic route fields to `Route` when project content needs parameterized pages. | [Dynamic routes](https://dioxuslabs.com/learn/0.7/essentials/router/routes/) |
